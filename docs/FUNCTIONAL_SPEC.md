# SSAFY 기능 명세서 초안

## 1. 문서 개요

### 1.1 목적
- 현재까지 확보한 대표 캡처/HTML/JSON을 기준으로 클론 코딩 전 단계의 **기능 명세서 초안**을 작성한다.
- 본 문서는 화면 단위 요구사항을 **기능 ID, 입력, 처리, 출력, 예외** 수준으로 세분화한 1차 분석 결과이다.

### 1.2 근거 범위
- 기준 산출물: `../ssafy_pages/capture-manifest.json`
- 안정적으로 확보된 대표 캡처: 22건
- 미확보 상세 화면: 2건
  - Quest 상세
  - Survey 상세
- 학습자료 eBook/PDF 내부 뷰어는 진입 흐름까지만 확인했고, 뷰어 내 렌더링 화면은 후속 검증 대상으로 남아 있다.

### 1.3 작성 원칙
- **직접 확인된 기능**을 우선 명세한다.
- 상세 화면 미확보 영역은 과도한 추론을 피하고, 필요한 경우 `추론`으로 표기한다.
- 본 문서는 **클론 코딩 범위 정의용 초안**이며, 실제 개발 착수 전 상세 플로우/정책 확인이 추가로 필요하다.

### 1.4 주요 액터
- **학생(Student)**: 일반 로그인 사용자, 학습/제출/응답 중심 사용자
- **선생(Instructor)**: 강의/평가/피드백 중심 사용자
- **운영자(Manager)**: 공지/문의/게시판/출결 처리 중심 사용자
- **관리자(Admin)**: 시스템 정책/권한/코드 관리 사용자
- **시스템(System)**: 인증, 조회, 상태 변경, 검증, 알림 제공 주체

### 1.5 역할 정책 메모
- revised schema 기준 전역 역할은 `student`, `instructor`, `manager`, `admin` 4개이다.
- 문서 설명상 권한 수준은 `학습자 / 실무 운영자 / 시스템 관리자` 3레벨로 묶어 설명할 수 있다.
- 반/수업 맥락 역할은 `user_class_enrollments.member_role_code`에서 별도 관리될 수 있다.

| 항목 | 내용 |
|---|---|
| 기능명 | 로그인 |
| 액터 | 교육생 |
| 목적 | 아이디/비밀번호 기반으로 시스템에 인증한다. |
| 사전조건 | 로그인하지 않은 상태여야 한다. |

| 관련 엔티티 | User, Session |

## 3. 공통 비즈니스 규칙 초안

### BR-001 인증 필요 화면 보호
- 로그인 후 화면은 세션이 유효한 사용자만 접근 가능해야 한다.

### BR-002 민감 기능 재인증
- 회원정보 수정 등 민감 기능은 현재 비밀번호 재확인을 요구해야 한다.

### BR-003 상태 기반 UI 노출
- Quest, 설문, 출결, 알림 등은 현재 상태에 따라 버튼/배지/진입 가능 여부가 달라져야 한다.

### BR-004 목록 중심 구조
- 대부분 모듈은 `목록 -> 상세 -> 작성/수정/응답` 흐름을 따른다.

### BR-005 사용자 개인화
- 출결, 알림, 설문, Quest, 회원정보는 로그인 사용자 기준 데이터만 노출해야 한다.
