# 역할/권한 정책표 (Role Matrix)

## 문서 목적
이 문서는 `docs/revised_schema_mysql8.sql`의 `USER_ROLE`과 `CLASS_MEMBER_ROLE` 정의를 바탕으로 **권장 역할 체계와 권한 정책**을 정리한 문서입니다.

## 역할 설계 결론
- **전역 역할(Role)은 4개**로 구분하는 것이 적절합니다.
  - `student`
  - `instructor`
  - `manager`
  - `admin`
- 다만 설명과 운영 정책에서는 **3단계 권한 레벨**로 묶는 것이 이해와 운영에 유리합니다.
  - Level 1: 학습자 (`student`)
  - Level 2: 실무 운영자 (`instructor`, `manager`)
  - Level 3: 시스템 관리자 (`admin`)

## revised schema 근거
- `USER_ROLE`: `student`, `instructor`, `manager`, `admin`
- `CLASS_MEMBER_ROLE`: `student`, `instructor`, `mentor`, `manager`
- 사용자 전역 역할은 `users.role_code`에서, 반/수업 맥락 역할은 `user_class_enrollments.member_role_code`에서 관리됩니다.

## 전역 역할 정의

| 역할 | 코드 | 설명 |
|---|---|---|
| 학생 | `student` | 학습/제출/응답/조회 중심 사용자 |
| 선생 | `instructor` | 강의/평가/피드백 중심 사용자 |
| 운영자 | `manager` | 운영/문의/공지/출결 처리 중심 사용자 |
| 관리자 | `admin` | 시스템 전체 정책/계정/권한 관리 사용자 |

## 권한 레벨 정의

| 레벨 | 포함 역할 | 설명 |
|---|---|---|
| Level 1 | `student` | 학습자 권한 |
| Level 2 | `instructor`, `manager` | 실무 운영 권한 |
| Level 3 | `admin` | 시스템/정책 최고 권한 |

## 운영 원칙

### 학생 (`student`)
- 본인 기준 열람/제출/응답/문의 등록 중심
- 타 사용자 관리 권한 없음

### 선생 (`instructor`)
- 학습 운영 중심
- 커리큘럼/학습자료/퀘스트/반 단위 출결 조회 및 일부 처리 가능
- 시스템 정책 관리 권한은 없음

### 운영자 (`manager`)
- 운영 처리 중심
- 공지/게시판/문의/출결/사용자 상태 일부 관리 가능
- 강의 작성/평가 권한은 서비스 정책에 따라 선택적으로 허용

### 관리자 (`admin`)
- 전역 최고 권한
- 코드, 역할, 정책, 접근 범위, 계정 상태까지 관리

## 설계 의견
- **선생과 운영자는 분리하는 것이 맞습니다.**
- 둘은 동일한 실무 레벨(Level 2)로 묶을 수는 있지만, 책임 영역이 다릅니다.
  - `instructor`: 교육 콘텐츠와 평가 중심
  - `manager`: 공지/문의/운영 처리 중심
- 따라서 **4역할 + 3권한레벨** 구조가 가장 설명력과 확장성이 좋습니다.

## 향후 확장 권고
현재 revised schema는 역할 코드 기반 구조입니다. 운영 정책이 더 복잡해지면 아래와 같은 RBAC 확장을 검토할 수 있습니다.
- `permissions`
- `role_permissions`
- `user_role_assignments` 또는 `admin_scope_assignments`

## 결론
현재 구조에서는 **학생 / 선생 / 운영자 / 관리자** 4개 역할로 유지하는 것이 맞고, 문서/정책 설명에서는 이를 **3단계 권한 레벨**로 묶어 관리하는 방식이 가장 실용적입니다.
