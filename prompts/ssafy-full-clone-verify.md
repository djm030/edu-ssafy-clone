너는 최종 검증 책임자다.

목표:
SSAFY 풀 클론 프로젝트가 실제로 완료되었는지 검증한다.
단순히 파일이 존재하는지 보지 말고, 기능 완성도와 실행 가능성을 기준으로 판단한다.

검증 대상:

- 코드
- DB schema
- API
- Frontend 화면
- API 연동
- 테스트
- Docker/env
- README
- docs/progress.md
- docs/architecture.md
- docs/api-summary.md
- docs/test-report.md
- docs/remaining-work.md

검증 절차:

1. 저장소 구조를 확인한다.
2. 최근 커밋 목록을 확인한다.
3. 문서와 실제 코드가 일치하는지 확인한다.
4. Backend 실행 가능 여부를 확인한다.
5. Frontend 실행/빌드 가능 여부를 확인한다.
6. 테스트 또는 smoke test를 실행한다.
7. 핵심 기능 체크리스트를 하나씩 검증한다.
8. 실패한 항목은 원인과 수정 방향을 기록한다.
9. 테스트 하네스가 부족하면 최소 smoke test를 추가한다.
10. 재검증한다.

판정 기준:
각 항목을 다음 중 하나로 분류한다.

- PASS: 실제 구현 및 검증 완료
- PARTIAL: 일부 구현됐지만 부족함
- FAIL: 구현 또는 검증 실패
- UNKNOWN: 확인 근거 부족

최종 산출물:
docs/final-verification.md 파일을 작성하거나 갱신한다.

1. 최종 검증 요약

4. 기능별 PASS/PARTIAL/FAIL/UNKNOWN 표
5. 발견한 문제

8. 최종 판단

중요:

- 실패한 테스트를 무시하지 않는다.
- 문서만 보고 완료 처리하지 않는다.
- 실제 실행 또는 빌드 근거를 남긴다.
- 완료 기준을 만족하지 못하면 완료라고 하지 않는다.
- 수정 가능한 문제는 직접 수정하고 커밋한다.
- 수정 후 다시 검증한다.
